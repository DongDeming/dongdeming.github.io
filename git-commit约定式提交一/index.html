git-commit约定式提交一

发表于 2020-10-27 | 分类于 前端构建 |

约定式提交规范的带来的好处

1. 通过快速的浏览提交记录就能知道代码改动的关键信息
2. 方便过滤或查找快速定位某些提交信息
3. 可以从commit快速生成 `change log`

Conventional Commits 规范 （Angular 规范 的延伸）

  1. 提交信息格式

    参数 说明
    type 改动的类型标记，有固定的一组单词规范
    scope 改动的范围（上下文），多个范围用 / \ , 分隔。比如数据层、控制层、视图层，改动页面等，视项目不同而不同。
    subject 简短描述，不超过50个字符。
    body 详细描述，可以分成多行。
    1. 语法使用第一人称现在时
    2. 应当说明变动的动机以及与以前行为的对比
    footer 仅描述两种情况
    1. 不兼容的变动： 如果当前代码与上一个版本不兼容，则 Footer 部分以BREAKING CHANGE开头，后面是对变动的描述、以及变动理由和迁移方法。
    2. 关闭 Issue： Closes #234
    type参数详解 说明
    build 影响构建系统或外部依赖项的更改(例如 glup，webpack，rollup 的配置等)的提交
    ci 主要目的是修改项目继续集成流程(例如 Travis，Jenkins，GitLab CI，Circle等)的提交
    chore 杂项，其他
    docs 文档相关
    feat 新增功能
    fix 修复bug
    perf 新能优化
    refactor 重构
    revert 于撤销以前的 commit，则必须以revert:开头，后面跟着被撤销 Commit 的 Header。 Body为 This reverts commit <hash>.
    style 非样式修改，而是代码的格式化等（不影响代码运行的变动）
    test 测试相关
