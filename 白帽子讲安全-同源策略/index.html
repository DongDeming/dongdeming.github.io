白帽子讲安全-同源策略

发表于 2022-02-14 | 分类于 读书笔记 |

同源策略

是提供的一种保障用户信息安全的一种约定，是浏览器最核心的安全功能。同源策略的源由协议, 主机, 端口组成的三元组确定。三元组相同则同源。

资源的跨源访问约束

  • 跨域的写操作（cross-origin writes）: 一般是被允许的，比如链接，重定向以及表单提交。（场景：更多的体现在链接参数的填写上，表单不算XMLHttpRequest）
  • 跨域的资源嵌入（cross-origin embedding） 一般是被允许的。（场景：js, css 的引入）
  • 跨域的读操作（cross-origin reads）一般是不被允许的。
    • window
      • 允许对某些属性方法的访问
    • DOM
      • 限制了对“document”对象的读取或设置某些属性；
    • 数据存储访问
      • 访问存储在浏览器中的数据，如 localStorage 和 IndexedDB，是以源进行分割。每个源都拥有自己单独的存储空间，一个源中的 JavaScript 脚本不能对属于其它源的数据进行读写操作。
      • 不管使用哪个协议（HTTP/HTTPS）或端口号，浏览器都允许给定的域以及其任何子域名(sub-domains) 访问 cookie。当你设置 cookie 时，你可以使用 Domain、Path、Secure、和 HttpOnly 标记来限定其可访问性。
    • XMLHttpRequest 和 Fetch API 遵循同源策略。
      • 允许 Web 应用服务器进行跨源访问控制，从而使跨源数据传输得以安全进行。
      • 可以使用 CORS 来允许跨源访问返回值
      • 请求是被允许的，如表单的请求
嵌入资源 说明
iframe 通常允许跨源嵌入（取决于X-Frame-Options），但不允许跨源读取
css 通常允许跨源嵌入，但需要设置正确的Content-Type
images 通常允许跨源嵌入，但不允许读图片信息（如将图片渲染进canvas，但是图片渲染后的宽高可以从DOM节点获得）
多媒体 通常允许跨源嵌入
script 通常允许跨源嵌入，但无法捕获语法错误

同源策略的规避

  • document.domain 脚本可以将 document.domain 的值设置为其当前域或其当前域的父域
    • 任何对document.domain的赋值操作，包括 document.domain = document.domain 都会导致端口号被重写为 null 。
    • 必须在他们双方中都进行赋值，以确保端口号都为 null 。
  • 片段识别符#fragment 如果只是改变片段标识符，页面不会重新刷新。
  • window.name属性：无论是否同源，只要在同一个窗口（tab）里，前一个网页设置了这个属性，后一个网页可以读取它。
  • window.postMessage: 跨文档通信 API
  • ajax
    • JSONP: 网页通过添加一个<script>元素，向服务器请求JSON数据，这种做法不受同源政策限制；服务器收到请求后，将数据放在一个指定名字的回调函数里传回来。
    • Websocket该协议不实行同源政策，只要服务器支持，就可以通过它进行跨源通信。
    • CORS是跨源资源分享，相比JSONP只能发GET请求，CORS允许任何类型的请求。
